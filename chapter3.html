🔣 整數表示

📊 基本概念

無符號整數

只能表示正數和零

n位元範圍： 0 到 2ⁿ-1
例子(4位元)： 0000(0) 到 1111(15)

有符號整數

可以表示正數、負數和零

n位元範圍： -2ⁿ⁻¹ 到 2ⁿ⁻¹-1
例子(4位元)： 1000(-8) 到 0111(7)

🔢 有符號整數的兩種表示方法

電腦需要用二進制表示正數和負數，主要有兩種方法：

方法一：符號及值 (Sign and Magnitude)

🔧 運作原理
  • • 第一位元：符號位（0=正數，1=負數）
  • • 其餘位元：表示數值大小
  • • 計算：符號位=0則為正，符號位=1則為負
📋 4位元完整例子
正數：
0000₂ = +0₁₀
0011₂ = +3₁₀
0101₂ = +5₁₀
0111₂ = +7₁₀
負數：
1000₂ = -0₁₀ ⚠️
1011₂ = -3₁₀
1101₂ = -5₁₀
1111₂ = -7₁₀
❌ 問題： 有兩個零的表示（0000₂=+0, 1000₂=-0），浪費一個組合

方法二：二進制補碼 (Two's Complement) ⭐

🔧 運作原理
  • • 最左位元：負權重位元（-2ⁿ⁻¹）
  • • 其餘位元：正權重位元
  • • 計算：-aₙ₋₁×2ⁿ⁻¹ + Σ(aᵢ×2ⁱ)
📋 4位元完整例子
負數：
1000₂ = -8₁₀
1001₂ = -7₁₀
1010₂ = -6₁₀
1011₂ = -5₁₀
1100₂ = -4₁₀
1101₂ = -3₁₀
1110₂ = -2₁₀
1111₂ = -1₁₀
非負數：
0000₂ = 0₁₀
0001₂ = 1₁₀
0010₂ = 2₁₀
0011₂ = 3₁₀
0100₂ = 4₁₀
0101₂ = 5₁₀
0110₂ = 6₁₀
0111₂ = 7₁₀
✅ 優點： 只有一個零、範圍更大（-8到+7）、計算簡單、現代標準

📊 兩種方法比較

特性 符號及值 二進制補碼
零的表示 兩個 (0000, 1000) 一個 (0000)
4位元範圍 -7 到 +7 -8 到 +7
計算複雜度 需要特殊處理 簡單統一
現代使用 少見 標準 ⭐

⚠️ 溢出誤差

📖 定義

當計算結果超出位元數可以表示的範圍時，就會發生溢出誤差。這些情況下，計算結果便會出錯。

無符號整數溢出例子 (4位元)

加法溢出：
9₁₀ + 9₁₀ = 1001₂ + 1001₂
= 10010₂ (5位元) → 0010₂ = 2₁₀ ❌
(正確應該是18₁₀，但超出4位元範圍)
減法溢出：
10₁₀ - 12₁₀ = 1010₂ - 1100₂
無法用無符號整數表示負數 ❌

有符號整數溢出例子 (4位元二補碼)

正數溢出：
3₁₀ + 5₁₀ = 0011₂ + 0101₂
= 1000₂ = -8₁₀ ❌
(兩個正數相加得到負數)
負數溢出：
(-4)₁₀ - 6₁₀ = 1100₂ + 1010₂
= 10110₂ → 0110₂ = 6₁₀ ❌
(兩個負數運算得到正數)

💡 重要提醒： 有多出的位元不一定代表溢出。只有當計算結果超出表示範圍才是溢出誤差。

🔤 字符編碼與條碼

📝 字符編碼系統

ASCII

• 英語字符編碼

• 7位元 = 128個字符

• 每個字符約1字節

中文編碼

• 大五碼：繁體中文 (2字節)

• 國標碼：簡體中文 (2字節)

• 兩者不能互換

Unicode (統一碼)

• 支援多種語言

• 可變長度編碼 (1-4字節)

• UTF-8, UTF-16格式

🧮 字符計算器

📱 條碼系統

📊 一維條碼

特點：
  • • 條形和空格寬度重要
  • • 只能從0°或180°掃描
  • • 最多約50個字符
應用： 商品條碼、圖書ISBN

📱 二維碼 (QR Code)

優勢：
  • • 任何方向掃描
  • • 高達30%錯誤修復
  • • 可存儲1000+字符
  • • 可部分遮擋仍可讀取
應用： URL連結、聯絡資訊、WiFi設定

💿 模擬與數碼數據

📊 數據類型比較

🌊 模擬數據

表示方式： 連續值變化

例子： 聲波、手寫文字、油畫

儲存媒體： 錄音帶、底片、黑膠唱片

特點：
  • • 壓縮能力低
  • • 數據分析能力低
  • • 需要手動更新

💾 數碼數據

表示方式： 離散符號 (0和1)

例子： 文字檔案、數碼照片、MP3

儲存媒體： 硬碟、USB、SSD

特點：
  • • 壓縮能力高
  • • 數據分析能力高
  • • 可自動更新計算
